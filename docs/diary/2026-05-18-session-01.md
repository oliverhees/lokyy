## Decisions + Rationale

### D26 — In-process cron in lokyy-mcp, NICHT Hermes-cron-bridge
**Rationale**: Hermes-cron-Integration hätte gebraucht: shared volume mount für scripts, per-Dashboard shell-skript-deployment, cleanup-script bei delete, edit-script bei schedule-change, docker-exec-Permission, sync mit `hermes cron list`. Sehr viele moving parts. Statt dessen: single setInterval-tick im lokyy-mcp container, scant alle 60s das `/app/data/dashboards/*` directory, vergleicht Schedule gegen aktuelles minute-hour-dom-mon-dow, ruft `invokeTool('run_now')`. Same outcome, ~120 LOC, zero docker-Coupling. Reversible: wenn jemand später Hermes-cron-Sync braucht, kann ein neuer Slice einen Bridge-Hook adden.

### D27 — Cron de-dup window 50s
**Rationale**: Tick alle 60s aber cron-match feuert wenn `now.minute matches expr`. Wenn Tick zufällig 2× im selben Minute-Slot landet (jitter), würde der Job 2× feuern. lastFiredAt-Check mit 50s window blockt das.

### D28 — Cron-Job-State über Reloads stabil
**Rationale**: Bei jedem Tick wird die Job-Liste vom Disk neu geladen (für neue Dashboards/Schedule-Edits). Aber lastFiredAt wäre dann zurückgesetzt → könnte mehrfach feuern. Fix: vor dem Reload die existing lastFiredAt in einer Map merken und nach Reload re-attachen.

### D29 — Bun-parser-Workaround
**Rationale**: Bun's TS-parser stolpert über backticks in JSDoc-Kommentaren wenn der Datei auch echte template-strings enthält. Ohne fix kein build möglich. Comment ohne backticks geschrieben.

## Phase-4.6 — Chat-Wizard für Dashboard-Erstellung (LLM-driven)

### Goal
Single-Field-Dialog durch echten Multi-Turn-Chat ersetzen. LLM (Hermes /v1/chat/completions) versteht den User-Intent konversationell und extrahiert eine strukturierte Spec (intent + schedule + title) wenn genug Info da ist.

### Decisions

**D30 — TZ-Quickfix BEFORE Chat-Wizard build**
**Rationale**: Oliver hat während des Phase-4.5-Mergens einen 12-Uhr-Cron-Test gemacht. Container war UTC, sein "12" wäre erst um 14:00 Berlin gefeuert (silent misfire). Fix: `TZ=Europe/Berlin` in lokyy-mcp env, konfigurierbar via `LOKYY_CRON_TZ`. 3-Zeilen-Commit gefühlt vor Chat-Wizard-Build, damit Olivers Test live verifizierbar war.

**D31 — Stateless Chat-Tool, FE hält History**
**Rationale**: Backend-State-Mgmt für Multi-Turn-Sessions würde Capability-Tokens oder Cookie-mapping brauchen. Stateless ist viel simpler: FE sendet bei jedem Turn die volle messages[]-history mit, Backend ruft LLM, returnt nächste Antwort. Cost: jedes Turn schickt die vorherige Konversation mit. Wert: einfache REST-API, kein Session-State auf Backend.

**D32 — JSON-im-Codeblock als "Ready"-Signal**
**Rationale**: LLM unterscheidet "konversationell antworten" vs "fertig, hier ist die Spec" durch Output-Form. System-Prompt instruiert: "wenn ready, antworte mit AUSSCHLIESSLICH einem JSON-Codeblock". Parser sucht nach ```json…```; wenn `ready:true` enthalten + alle Felder valid → kind:'spec', sonst kind:'message'. Robust gegen LLM-Schwankungen, einfacher als function-calling.

**D33 — Hermes-System-prompt auf Deutsch, knapp + freundlich**
**Rationale**: User sind deutschsprachig. LLM antwortet sonst auf englisch. Plus: Limit auf "1-3 Turns" verhindert dass LLM endlos weiter fragt. In meinem Test antwortete der LLM oft schon im ersten Turn mit Spec, weil User-Intent klar genug war ("KI-News täglich um 9 Uhr, Titel AI Pulse").
