## Phase-4 Sub-Slice 1 — lokyy-mcp skeleton + auth

### Goal
ISCs 82–84: container + MCP protocol skeleton + bearer-token auth, all behind lokyy-net (no public surface).

### Decisions taken

**D6 — Custom SSE-Transport instead of SDK's SSEServerTransport**
**Rationale**: \`@modelcontextprotocol/sdk\`'s SSEServerTransport assumes Node+Express \`(req, res)\` style. Bun+Hono use the Web Response API. Monkey-patching the SDK class via \`Object.create(...prototype)\` was fragile (\`server.connect\` crashed). Re-implementing the \`Transport\` interface from scratch is ~50 lines of TS, much more transparent. Decided to ship the custom transport.

**D7 — POST + GET endpoint pair (\`/mcp\` SSE, \`/mcp/messages\` POST) instead of single streaming HTTP endpoint**
**Rationale**: That's the MCP-SSE canonical pattern. Client opens long-lived SSE, server emits \`endpoint\` event with a session-scoped POST URL. Sessions tracked in an in-memory \`Map<sessionId, Transport>\`. Survives single-replica restarts (each restart = clients reconnect).

**D8 — \`LOKYY_SYSTEM_SECRET\` generated via \`openssl rand -hex 32\` in \`.env.local\` only**
**Rationale**: 32-byte hex = 256-bit entropy, plenty for a long-lived shared secret. Never committed; \`.env.example\` shows the variable but value is empty. Auth-comparison is constant-time to avoid length-leak.
