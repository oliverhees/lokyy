## Decisions + Rationale

### D1 — Phase-2c merge first, Phase-2b rebase on top
**Rationale**: cleaner base, no rebase-churn for Phase-4 work that depends on Phase-2c's conversations layer.

### D2 — Phase-2b bug had two parts
**Rationale**: (1) supervisor didn't distinguish 'stopped' vs 'reaped' containers → added \`hermes-missing\` event kind with 30min backoff. (2) The actual restart-404 was an **off-by-one in container name** — \`HERMES_CONTAINER_NAME=lokyy-hermes-1\` (compose-v2 default) but service has \`container_name: lokyy-hermes\` explicit. Both fixed; the missing-event detection is still correct and useful.

### D3 — Lokyy-MCP is a system-wide bus, not a dashboard feature
**Rationale**: Oliver's correction mid-brainstorm. MCP exposes platform-wide tools for all System Skills (DashboardBuilder, MemoryCurator, NotificationRouter, …). Dashboard-Skill is just the first consumer.

### D4 — System Skills = Lokyy-owned TypeScript code
**Rationale**: Token-leak risk eliminated (secret never crosses into Hermes), one audit surface, no dependency on Hermes feature-set. Lives in \`lokyy-mcp/src/system-skills/\`. Immutable from user.

### D5 — Capability-Tokens for User-Skill narrow access
**Rationale**: When a System Skill (e.g. DashboardBuilder) generates a User-Skill (e.g. Producer for that Dashboard) that needs to write back to a specific Lokyy state slice, Lokyy-MCP issues a scoped revocable token. Principle of least privilege: token covers exactly one operation on one target.
