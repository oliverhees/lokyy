## Phase-4 Sub-Slice 1 — lokyy-mcp skeleton + auth

### Goal
ISCs 82–84: container + MCP protocol skeleton + bearer-token auth, all behind lokyy-net (no public surface).

### Decisions taken

**D6 — Custom SSE-Transport instead of SDK's SSEServerTransport**
**Rationale**: \`@modelcontextprotocol/sdk\`'s SSEServerTransport assumes Node+Express \`(req, res)\` style. Bun+Hono use the Web Response API. Monkey-patching the SDK class via \`Object.create(...prototype)\` was fragile (\`server.connect\` crashed). Re-implementing the \`Transport\` interface from scratch is ~50 lines of TS, much more transparent. Decided to ship the custom transport.

**D7 — POST + GET endpoint pair (\`/mcp\` SSE, \`/mcp/messages\` POST) instead of single streaming HTTP endpoint**
**Rationale**: That's the MCP-SSE canonical pattern. Client opens long-lived SSE, server emits \`endpoint\` event with a session-scoped POST URL. Sessions tracked in an in-memory \`Map<sessionId, Transport>\`. Survives single-replica restarts (each restart = clients reconnect).

**D8 — \`LOKYY_SYSTEM_SECRET\` generated via \`openssl rand -hex 32\` in \`.env.local\` only**
**Rationale**: 32-byte hex = 256-bit entropy, plenty for a long-lived shared secret. Never committed; \`.env.example\` shows the variable but value is empty. Auth-comparison is constant-time to avoid length-leak.

## Phase-4 Sub-Slice 2 — Capability-Tokens (ISC-85)

### Goal
Privilege-Layer für User-Skills. System-Skills bleiben full-access; User-Skills (Producers) bekommen narrow-scoped, revocable tokens.

### Decisions

**D9 — Capability-Token-Format `Capability-<tokenId>-<secret>`**
**Rationale**: tokenId = 8 bytes hex (16 chars) = ID-portion für Audit-Trail, secret = 24 bytes hex (48 chars) für Entropie. Prefix `Capability-` macht eindeutig erkennbar im Auth-Middleware. Volle Bearer-string wird nur bei Issue-Response zurückgegeben, danach in `capabilities.json` nur als opaque field.

**D10 — Bearer im List-Response versteckt**
**Rationale**: `GET /admin/capabilities` zeigt tokenId + scope + target + audit-Felder, aber NICHT den vollen bearer-string. Caller bekommt bearer einmalig beim Mint und ist verantwortlich für sichere Weitergabe (ENV-injection in den Consumer-Skill). Verhindert "Token-Lookup-Angriff" über kompromittierten Admin-Account.

**D11 — `requireSystem` separate Middleware statt nested-call**
**Rationale**: Erste Version war `requireSystem` calling `requireBearer` mit no-op next + post-check des Principal. Hono's Middleware-Modell unterstützt das nicht sauber (Response-Write-Reihenfolge). Re-implementiert als standalone middleware: System-bearer-only, 403 für alles andere (auch valid Capability-tokens). Cleaner trust-boundary.

**D12 — Audit-Log als JSONL in shared volume**
**Rationale**: Gleiches Pattern wie `activity.jsonl` (Phase-2b) — POSIX-atomic append, no auth-juggling, deterministisch für tests. Events: `issue`, `revoke`, `deny`, `use`. Audit-trail kann später als System-Skill ausgelesen werden für UI-Anzeige.

## Phase-4 Sub-Slice 4 — save_data Tool (ISC-92)

### Goal
Producer-Counterpart zu create_via_builder. Capability-scoped, scope+target-enforced. Schließt die Closed-Loop Pipeline (create → produce → save → display).

### Decisions

**D17 — HTTP one-shot path `/tools/:name/invoke` neben MCP-SSE**
**Rationale**: Capability-Tokens am `/admin/tools/...` Pfad gingen nicht (system-only-gate). Saubere Lösung: separate `tools-http.ts` Router mit `requireBearer` (akzeptiert beide Bearer-Typen) statt `requireSystem`. Per-tool privilege-Check passiert in `invokeTool()`. Vorteile:
- Stateless one-shot call statt MCP-SSE-handshake (für Producer mit cron-style einmalige Aufrufe ideal)
- Verify-scripts können beide Principals testen
- Sub-Tools-/admin-only-Endpunkt wird redundant → entfernt

**D18 — Capability-Check VOR existence-check in save_data**
**Rationale**: Wenn `ensureDashboardExists` zuerst läuft, leakt der Server timing-/error-message-mäßig ob dashboard-id existiert. Capability mit anderem Target könnte so andere IDs probieren. Reorder: Scope+Target-Check ZUERST, dann existence. Capability-User sehen exakt `target_mismatch` egal ob das andere Dashboard existiert oder nicht.

**Rationale**: Iframe könnte vor dem Parent-Daten-Loop fertig laden. Iframe sendet `lokyy:dashboard:ready` an `window.parent`, Parent antwortet mit `lokyy:dashboard:data`. Parent pusht auch wenn data sich ändert (Datum-Selektion). Origin-Check beim Senden auf `window.location.origin`.

**Rationale**: Dashboard-ID ist `ki-news-<hex>` (technische ID), aber User sieht "KI-News" als Titel. Server-side gemappt damit FE keine Template-Logik kennt. Erweiterbar wenn Custom-Templates kommen.

## Phase-4.2 — Real-World Producer mit echten Daten

### Goal
"Jetzt laufen" Button funktioniert wirklich. KI-News-Dashboard zeigt echte AI/ML-Stories aus Hacker News, refresh-bar per Klick. Real-world-Demo des kompletten Loops.

### Decisions

**D23 — Producer als built-in TS-Module statt Hermes-Skill-Loader**
**Rationale**: Hermes-Skill-Format laden + ausführen ist ein eigenes Sub-System (das ein eigenständiger Skill-runtime sein müsste). Pragmatischer Bridge: built-in TypeScript-Producers im `lokyy-mcp/src/system-skills/Producer/` Modul, ein Handler pro Template-key. Producer.skill.md bleibt als Spec-Dokumentation für die Zukunft. Vorteile: kein neuer Runtime, schneller fertig, deterministische Tests.

**D24 — Hacker News API als Daten-Quelle für KI-News**
**Rationale**: Free, no auth, JSON, stable. Top-stories filtered by AI/ML keywords (whole-word match für 2-3 letter keys wie "ai" damit "rain" nicht matched). Top 10 nach Score. Production-ready ohne API-Keys.

**D25 — postMessage targetOrigin = `'*'` (statt `window.location.origin`)**
**Rationale**: Iframe ist mit `sandbox="allow-scripts"` ohne `allow-same-origin` gemountet → opaque/null origin. Posting mit konkretem Origin wird vom Browser blockiert. `'*'` ist hier sicher, weil:
1. Iframe content kommt aus unserem eigenen Backend, ist not user-generated
2. Wir kontrollieren beide Seiten der postMessage-Schnittstelle
3. Browser-Sicherheit: `'*'` heißt "any recipient", aber unser data payload enthält keine sensitiven Secrets
